# Credit Docs Checker

REST API сервис проверки пакетов документов по льготным кредитным программам (federal / regional) с сохранением истории проверок в PostgreSQL.

## Быстрый старт

Требуется Docker с Compose v2.

```bash
docker compose up --build
```

Сервис поднимется на `http://localhost:8000` (миграции Alembic применяются автоматически при старте).

- Swagger UI: http://localhost:8000/docs
- Healthcheck: http://localhost:8000/health

### Пример запроса

```bash
curl -X POST http://localhost:8000/api/checks \
  -F "program=federal" \
  -F "files=@договор_47.pdf" \
  -F "files=@спецификация.pdf" \
  -F "files=@счет_105.pdf" \
  -F "files=@акт_приемки.pdf"
```

```bash
curl http://localhost:8000/api/checks
curl http://localhost:8000/api/checks/{check_id}
```

## Запуск тестов

Локально (нужен [uv](https://docs.astral.sh/uv/)):

```bash
uv sync
uv run pytest
```

Тесты - чистые unit-тесты (классификация типа документа и формирование итогового статуса), БД не требуется.

## API

| Метод | Путь | Описание |
|---|---|---|
| POST | `/api/checks` | Принимает файлы (multipart/form-data, поле `files`) и `program` (federal / regional), запускает проверку, сохраняет результат в БД и возвращает его |
| GET | `/api/checks` | Список всех проверок: id, дата, программа, статус, количество документов |
| GET | `/api/checks/{id}` | Полный результат конкретной проверки |

HTTP-статусы: `201` - проверка создана, `400` - нет файлов / файл без имени, `404` - проверка не найдена, `422` - невалидные параметры (например, неизвестная программа).

### Логика проверки

1. Тип документа определяется по имени файла: договор (contract), спецификация (specification), счёт (invoice), акт/УПД (act). Нераспознанное имя - warning.
2. Комплектность пакета: для `federal` обязательны договор, спецификация, счёт, акт; для `regional` - договор, счёт, акт. Отсутствие обязательного документа - error.
3. Формат файла: только PDF, DOCX, JPG, PNG - иначе error.
4. Размер файла: более 20 МБ - error.
5. Итоговый статус: `approved` (нет ошибок), `rejected` (есть хотя бы одна ошибка), `check_in_progress` (проверка выполняется).

Поле `extracted` в ответе - точка интеграции AI-модуля извлечения реквизитов из содержимого документов; возвращается структура `{contractor, amount, date, subject}` с пустыми значениями до подключения модуля.

## Технологии и почему

- **Python 3.11 + FastAPI** - требование задания; async-фреймворк с автогенерацией OpenAPI и валидацией через Pydantic из коробки.
- **PostgreSQL + SQLAlchemy 2.0 (async, asyncpg)** - требование задания; современный декларативный маппинг с типизацией, полностью асинхронный доступ к БД.
- **Alembic** - версионируемые миграции схемы, применяются автоматически при старте контейнера.
- **Pydantic v2 + pydantic-settings** - валидация запросов/ответов и конфигурация из `.env`.
- **uv** - быстрый менеджер зависимостей с lock-файлом (воспроизводимые сборки локально и в Docker).
- **pytest** - unit-тесты бизнес-логики без внешних зависимостей.
- **Docker / docker compose** - весь стек поднимается одной командой.

## Архитектура

```
app/
├── main.py          # создание FastAPI-приложения, подключение роутеров
├── config.py        # настройки из .env (pydantic-settings)
├── database.py      # async engine, session factory, база ORM-моделей
├── models.py        # ORM: Check, Document, Issue (история проверок)
├── schemas.py       # Pydantic-схемы API, enum-ы, справочники программ
├── api/
│   └── checks.py    # HTTP-слой: эндпоинты /api/checks
└── services/
    ├── classifier.py     # определение типа документа по имени файла
    ├── validator.py      # правила проверки и формирование итогового статуса
    └── check_service.py  # оркестрация: проверка, сохранение файлов и записи в БД
```

Слои разделены: HTTP-слой (`api`) не содержит бизнес-логики, бизнес-правила (`services/classifier.py`, `services/validator.py`) - чистые функции без I/O (что позволяет тестировать их без БД), запись в БД и файловое хранилище изолированы в `check_service.py`.

Хранение данных:

- `checks` - проверка: программа, статус, причина отказа, извлеченные данные, дата
- `documents` - файлы пакета: имя, распознанный тип, размер, путь в хранилище
- `issues` - найденные нарушения: уровень (error / warning), сообщение

Загруженные файлы сохраняются в `UPLOAD_DIR/<check_id>/` (в Docker - именованный volume `uploads`).

## Безопасность

- Имена файлов санитизируются и в хранилище, и в ответах API (защита от path traversal и хранимых инъекций: `../../etc/passwd` -> `passwd`, управляющие символы вырезаются).
- Защита от DoS: файлы сверх лимита размера не читаются в память и не пишутся на диск (только фиксируется ошибка), ограничено число файлов на запрос (`MAX_FILES_PER_CHECK`) и длина имени (`MAX_FILENAME_LENGTH`); слишком большие/длинные запросы отклоняются с кодом 400.
- Дубликаты имён в пакете не перезатирают друг друга (файлы на диске уникализируются префиксом), пустые файлы отклоняются.
- Валидация входа: формат и размер файлов, программа - строго из перечисления, id проверки - строго UUID.
- SQL-инъекции исключены: только параметризованные запросы через SQLAlchemy ORM.
- Контейнер приложения работает от non-root пользователя; порт PostgreSQL наружу не публикуется.
- Секреты в репозиторий не попадают: `.env` в `.gitignore`, в `.env.example` только dev-значения.

Аутентификация и разграничение прав доступа (упомянуты в бизнес-кейсе как TO BE) в объем тестового задания (пункты 1-9) не входят. В продуктиве сервис живет за API-шлюзом банка: OAuth2/JWT, роли (специалист казначейства / руководитель / AI-модуль) на уровне шлюза и middleware.

## Переменные окружения (.env.example)

| Переменная | По умолчанию | Описание |
|---|---|---|
| `DATABASE_URL` | `postgresql+asyncpg://postgres:postgres@db:5432/docs_checker` | Строка подключения к PostgreSQL (asyncpg) |
| `POSTGRES_USER` | `postgres` | Пользователь PostgreSQL (контейнер db) |
| `POSTGRES_PASSWORD` | `postgres` | Пароль PostgreSQL (контейнер db) |
| `POSTGRES_DB` | `docs_checker` | Имя базы данных |
| `UPLOAD_DIR` | `uploads` | Каталог для хранения загруженных файлов |
| `MAX_FILE_SIZE_MB` | `20` | Максимальный размер одного файла, МБ |
| `ALLOWED_EXTENSIONS` | `pdf,docx,jpg,jpeg,png` | Разрешенные расширения файлов |

Для запуска через `docker compose up` файл `.env` не обязателен - используются значения по умолчанию. Для переопределения: `cp .env.example .env` и отредактировать.

## Локальный запуск без Docker

```bash
uv sync
cp .env.example .env  # поменять host db на localhost в DATABASE_URL
uv run alembic upgrade head
uv run uvicorn app.main:app --reload
```
